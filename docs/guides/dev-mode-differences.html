Este documento explica cómo el modo dev en Quarkus difiere de una aplicación de producción.

Introducción

Quarkus proporciona un modo de desarrollo (explicado con más detalle aquí y aquí) que ayuda enormemente durante el desarrollo, pero NUNCA debe usarse en producción.

Diferencias arquitectónicas

Dejando a un lado los conjuntos de características, la aplicación Quarkus que se ejecuta en modo dev difiere arquitectónicamente de la aplicación de producción (es decir, la que se ejecuta utilizando java -jar …​).

En el modo dev, Quarkus utiliza una jerarquía ClassLoader (explicada en detalle aquí) que permite la recarga en vivo del código del usuario sin requerir una reconstrucción y reinicio de la aplicación.

En una aplicación de producción, la mencionada infraestructura de carga de clases está totalmente ausente - hay un único ClassLoader construido a propósito que carga (casi) todas las clases y dependencias.

Características del modo de desarrollo

Siguiendo el mantra de proporcionar satisfacción a los desarrolladores, Quarkus proporciona una gran cantidad de características cuando el modo dev está activado. Las características más importantes son:

Recarga en vivo

Esta característica tan importante no necesita presentación y ya se ha mencionado en la sección de architectural differences.

Interfaz de usuario

Quarkus proporciona una UI muy útil accesible desde el navegador en /q/dev. Esta UI permite a un desarrollador ver el estado de la aplicación, pero también proporciona acceso a varias acciones que pueden cambiar ese estado (dependiendo de las extensiones que estén presentes). Ejemplos de estas operaciones son:

  • Modificación de los valores de configuración

  • Ejecución de scripts de migración de bases de datos

  • Borrado de cachés

  • Ejecución de operaciones programadas

  • Construir un contenedor

Páginas de error

En un esfuerzo por hacer que los errores de desarrollo sean muy fáciles de diagnosticar, Quarkus proporciona varias páginas de error detalladas cuando se ejecuta en modo dev.

Scripts de importación de bases de datos

La extensión quarkus-hibernate-orm ejecutará el script import.sql en src/main/resources cuando Quarkus se ejecute en modo dev. Se pueden encontrar más detalles aquí.

Servicios de desarrollo

Cuando se prueba o se ejecuta en modo dev, Quarkus puede incluso proporcionarle una base de datos de configuración cero fuera de la caja, una característica a la que nos referimos como Servicios Dev. Puede encontrar más información aquí.

Interfaz de usuario de Swagger

La extensión quarkus-smallrye-openapi expondrá la interfaz de usuario Swagger cuando Quarkus se ejecute en modo dev. Se puede encontrar información aquí.

GraphQL UI

La extensión quarkus-smallrye-graphql expondrá la UI de GraphiQL cuando Quarkus se ejecute en modo dev. Se pueden encontrar más detalles aquí.

IU de la Salud

La extensión quarkus-smallrye-health expondrá la interfaz de usuario de salud cuando Quarkus se ejecute en modo de desarrollo. Esta sección proporciona información adicional.

Simulacro de envío de correo

La extensión quarkus-mailer habilitará un servidor de correo simulado en memoria cuando Quarkus se ejecute en modo de desarrollo. Vea esto para más detalles.

gRPC

  • El servicio gRPC Reflection está activado por defecto en el modo dev. Esto le permite utilizar herramientas como grpcurl. En el modo de producción, el servicio de reflexión está desactivado. Puedes habilitarlo explícitamente usando quarkus.grpc-server.enable-reflection-service=true.

  • En el modo dev, quarkus.grpc.server.instances no tiene ningún efecto.

Otros

Puede haber otras propiedades de configuración (dependiendo de las extensiones añadidas a la aplicación) que no tienen efecto en el modo dev.

Implicaciones del rendimiento

En el modo dev, minimizar la huella en tiempo de ejecución de la aplicación no es el objetivo principal (aunque Quarkus todavía se inicia bastante rápido y consume poca memoria incluso en el modo dev) - el objetivo principal es permitir la alegría del desarrollador. Por lo tanto, se cargan muchas más clases y las operaciones en tiempo de compilación también tienen lugar cada vez que se realiza una recarga en vivo.

En cambio, en una aplicación de producción el objetivo principal de Quarkus es consumir la menor cantidad de memoria y arrancar en el menor tiempo posible. Así, cuando se ejecuta la aplicación de producción, las operaciones en tiempo de compilación no se realizan (por definición) y varias clases de infraestructura necesarias en tiempo de compilación no están presentes en tiempo de ejecución. Además, el ClassLoader construido a propósito que viene con el tipo de paquete fast-jar asegura que la búsqueda de clases se haga lo más rápido posible mientras se mantiene la mínima cantidad de jars en memoria.

Implicaciones de seguridad

Tal vez la razón más importante por la que las aplicaciones en modo dev no deberían ejecutarse en producción es que el modo dev permite leer información que podría ser confidencial (a través de la Dev-UI) al tiempo que da acceso a operaciones que podrían ser destructivas (ya sea exponiendo puntos finales que no deberían estar disponibles en la aplicación de producción o a través de la Dev-UI).

Ejecución nativa

Cuando se crea un ejecutable nativo (explicado en detalle aquí), siempre se construye a partir de una aplicación de producción.
